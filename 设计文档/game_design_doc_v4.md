# 《撤离者》游戏设计文档 v4.0

> 开发方案 | 单人独立游戏 | Unity 2D 俯视角 Roguelite 搜打撤
> 本版本新增：**地图随机生成详细设计** + **难度迭代系统**

## 一、项目概述

### 1.1 游戏简介

**《撤离者》** 是一款 2D 俯视角 Roguelite 搜打撤游戏。玩家选择起始特质进入程序生成的地图，在探索中不断获得随机强化、遗物和装备，构建出独特的 build。最终找到撤离点成功撤出，带回来的核心遗物可以永久解锁，成为后续局的优势。

### 1.2 一句话定位

每局都是不同的 build 体验，撤出来才算你的。

### 1.3 核心循环

```
选择特质 → 进入地图 → 局内成长（升级/遗物/事件）→ 撤离
                    ↓
              撤离成功 → 解锁永久遗物/解锁新特质
              撤离失败 → 本局遗物丢失
```

## 二、地图随机生成系统（详细设计）

### 2.1 生成算法：BSP 二叉空间分割

#### 2.1.1 算法概述

使用 **BSP（Binary Space Partitioning）** 算法将整张地图切割成若干区域，每个区域放置一个房间，再用走廊连接相邻房间。

```
整张地图（例如 80×60 格）
        ↓ 随机切割
  左半区域 | 右半区域
     ↓           ↓
 继续切割     继续切割
     ↓           ↓
  最终叶节点 = 每个房间的区域
```

#### 2.1.2 BSP 切割流程（伪代码）

```
function SplitNode(node, depth):
    if depth >= MAX_DEPTH or node太小:
        return  // 停止切割，这是叶节点（房间区域）

    // 随机选择切割方向
    if node.width > node.height * 1.25:
        direction = VERTICAL    // 宽大于高，竖切
    elif node.height > node.width * 1.25:
        direction = HORIZONTAL  // 高大于宽，横切
    else:
        direction = random(VERTICAL, HORIZONTAL)

    // 随机切割位置（在 40%~60% 之间，避免极端不均匀）
    splitPos = random(node.size * 0.4, node.size * 0.6)

    node.left  = new Node(左半区域)
    node.right = new Node(右半区域)

    SplitNode(node.left,  depth + 1)
    SplitNode(node.right, depth + 1)
```

#### 2.1.3 房间生成

每个叶节点区域内，随机生成一个房间（比区域略小，留出走廊空间）：

```
区域大小：例如 20×15 格
房间大小：在区域内随机，例如 12×9 ~ 16×12 格
房间位置：在区域内随机偏移（保证四周至少留 2 格走廊空间）
```

**房间尺寸约束：**

| 参数 | 值 |
|------|-----|
| 最小房间宽/高 | 8 格 |
| 最大房间宽/高 | 区域尺寸 - 4 格 |
| 房间内边距 | 至少 2 格（留走廊） |

#### 2.1.4 走廊连接

BSP 树从叶节点向上合并时，连接兄弟节点的房间：

```
function ConnectNodes(nodeA, nodeB):
    pointA = nodeA 房间内随机一点
    pointB = nodeB 房间内随机一点

    // L 形走廊（先横后竖，或先竖后横，随机选）
    if random() > 0.5:
        先从 pointA 横向走到 pointB.x
        再从该点纵向走到 pointB.y
    else:
        先从 pointA 纵向走到 pointB.y
        再从该点横向走到 pointB.x
```

**走廊宽度：** 固定 2 格（玩家可以并排通过）

#### 2.1.5 完整生成流程

```
1. 初始化地图（全部为墙壁）
2. BSP 切割整张地图 → 得到叶节点列表
3. 每个叶节点内生成一个房间（挖空地板）
4. 从 BSP 树根节点向下递归，连接兄弟节点（挖走廊）
5. 指定特殊房间（起始房间、撤离房间、商店等）
6. 在各房间内生成敌人、物资、事件
7. 生成完毕，玩家从起始房间出发
```

### 2.2 房间类型分配

#### 2.2.1 房间类型

| 房间类型 | 内容 | 出现权重 |
|---------|------|---------|
| 战斗房间 | 3~6 个敌人 + 清理后奖励选择 | 40% |
| 精英房间 | 1 个精英怪 + 高级奖励 | 15% |
| 商店房间 | 可购买遗物/恢复/强化 | 10% |
| 事件房间 | 随机事件（选择题） | 15% |
| 宝藏房间 | 1 个遗物（无战斗） | 10% |
| 混合房间 | 敌人 + 物资混合 | 5% |
| 撤离房间 | 包含撤离点（固定在末端） | 固定 1 个 |

#### 2.2.2 分配规则

```
1. 第一个房间（离起点最近）→ 固定为安全区（无敌人）
2. 最后一个房间（离起点最远）→ 固定为撤离房间
3. 每 4~5 个房间中，至少有 1 个精英房间
4. 商店房间不相邻出现（间隔至少 3 个房间）
5. 其余房间按权重随机分配
```

#### 2.2.3 房间连通性保证

- 所有房间必须可达（BSP 连接保证）
- 撤离房间只有 1 个入口（增加撤离时的压迫感）
- 宝藏房间可能有隐藏入口（需要找到隐藏走廊）

## 三、难度迭代系统（详细设计）

### 3.1 设计思路

难度迭代分为两个维度：

| 维度 | 说明 |
|------|------|
| **局内难度递进** | 同一局内，越深入地图越难 |
| **跨局难度提升** | 热度系统，玩家主动选择更高难度 |

---

### 3.2 局内难度递进

#### 3.2.1 房间深度系统

每个房间有一个**深度值**（Distance from Start），从起始房间（深度 0）到撤离房间（深度最大）。

```
起始房间（深度0）→ 普通房间（深度1~3）→ 精英房间（深度4~6）→ 撤离房间（深度最大）
```

#### 3.2.2 深度影响敌人强度

| 深度区间 | 敌人血量倍率 | 敌人伤害倍率 | 敌人数量 |
|---------|------------|------------|---------|
| 0~2（浅层） | ×1.0 | ×1.0 | 2~3 个 |
| 3~5（中层） | ×1.2 | ×1.15 | 3~5 个 |
| 6~8（深层） | ×1.5 | ×1.3 | 4~6 个 |
| 9+（极深层） | ×2.0 | ×1.5 | 5~7 个 |

#### 3.2.3 深度影响奖励品质

越深的房间，奖励品质越高：

| 深度区间 | 普通遗物概率 | 精良遗物概率 | 稀有遗物概率 |
|---------|------------|------------|------------|
| 0~2 | 80% | 18% | 2% |
| 3~5 | 60% | 35% | 5% |
| 6~8 | 40% | 45% | 15% |
| 9+ | 20% | 50% | 30% |

**设计意图：** 玩家面临风险收益抉择——越深越危险，但奖励越好。可以在浅层安全撤离，也可以冒险深入。

#### 3.2.4 局内时间压力（可选机制）

进入地图后，**120 秒**后开始触发"增援呼叫"：

```
0~120秒：正常探索
120秒后：每 30 秒刷新一批增援敌人（从地图边缘进入）
300秒后：增援频率加倍，敌人属性+20%
```

**目的：** 防止玩家无限磨蹭，保持紧张感。

---

### 3.3 跨局难度提升：热度系统

#### 3.3.1 热度系统概述

玩家完成首次撤离后解锁热度系统。每次进入地图前，可以自由选择热度等级（0~20）。

**核心设计原则：**
- 热度是**玩家主动选择**的，不是强制的
- 每个热度等级增加一个具体的难度修改器
- 每个热度等级对应一个具体的奖励提升

#### 3.3.2 热度修改器列表

| 热度等级 | 难度修改器 | 奖励提升 |
|---------|----------|---------|
| 热度 1 | 敌人生命值 +20% | 金币掉落 +15% |
| 热度 2 | 敌人伤害 +15% | 遗物掉落率 +5% |
| 热度 3 | 敌人数量 +1（每个房间） | 金币掉落 +20% |
| 热度 4 | 撤离等待时间 +5 秒 | 精良遗物概率 +10% |
| 热度 5 | 增援呼叫提前至 90 秒 | 稀有遗物概率 +5% |
| 热度 6 | 精英怪必出（每个精英房间 2 个） | 传说遗物概率 +3% |
| 热度 7 | 敌人生命值再 +20% | 金币掉落 +25% |
| 热度 8 | 敌人伤害再 +15% | 遗物掉落率 +10% |
| 热度 9 | 商店物品价格 +30% | 起始遗物品质提升 |
| 热度 10 | 死亡惩罚：丢失所有遗物（原本只丢失本局） | 解锁专属成就 |
| 热度 11~20 | 叠加前面的修改器，强度递增 | 奖励继续提升 |

#### 3.3.3 热度叠加规则

热度修改器**全部叠加**，不是替换：

```
热度 3 = 热度1修改器 + 热度2修改器 + 热度3修改器
       = 敌人生命+20% + 敌人伤害+15% + 敌人数量+1
```

#### 3.3.4 热度记录

- 记录玩家历史最高热度
- 每次超越个人最高热度，解锁专属称号
- 热度 10 以上解锁特殊结局动画

---

### 3.4 难度与 Build 的交互

热度系统不只是数值堆叠，某些热度修改器会改变最优 build 策略：

| 热度修改器 | 对 build 的影响 |
|----------|--------------|
| 敌人数量 +1 | 范围伤害遗物价值提升（爆裂弹、散射） |
| 撤离等待 +5 秒 | 时间裂隙遗物（撤离-50%）价值提升 |
| 商店价格 +30% | 金币相关遗物（掠夺者、贪婪者契约）价值提升 |
| 增援提前 | 移速/冲刺遗物价值提升（快速清场） |

**设计意图：** 不同热度等级下，最优 build 不同，增加策略深度。

---

### 3.5 难度与地图生成的联动

热度等级影响地图生成参数：

| 热度区间 | 地图变化 |
|---------|---------|
| 热度 0~4 | 标准地图（8~12 个房间） |
| 热度 5~9 | 地图房间数 +2，深层房间比例提升 |
| 热度 10~14 | 地图房间数 +4，隐藏房间概率 +20% |
| 热度 15~20 | 地图房间数 +6，撤离房间有 2 个守卫 Boss |

- 永久遗物可在后续局中作为起始选项选择
- 每次撤离后随机抽取 3 个永久遗物，选择 1 个作为起始遗物
- 鼓励玩家多次游玩，积累遗物库

## 六、撤离系统

### 6.1 撤离流程

1. 到达撤离房间
2. 击败守卫的 Boss（可选，击败有额外奖励）
3. 等待 8~12 秒（可被遗物缩短）
4. 等待期间敌人持续刷新
5. 倒计时结束，成功撤离

### 6.2 撤离结果

| 结果 | 条件 | 后果 |
|------|------|------|
| 成功撤离 | 等待结束 | 保留所有遗物，解锁永久遗物 |
| 死亡 | 等待期间被击杀 | 遗物全部丢失 |
| 强制撤离 | 未到撤离点，时间耗尽 | 只保留 1 个遗物（自选） |

### 6.3 撤离奖励

| 撤离类型 | 奖励 |
|---------|------|
| 普通撤离 | 保留遗物 |
| 完美撤离（HP > 80%） | 额外获得 1 个随机精良遗物 |
| Boss 击杀撤离 | 额外解锁 1 个永久遗物选项 |

### 7.2 核心类设计

#### BSPGenerator

```csharp
public class BSPGenerator : MonoBehaviour
{
    [Header("地图参数")]
    public int mapWidth = 80;
    public int mapHeight = 60;
    public int maxDepth = 4;
    public int minRoomSize = 8;

    // 生成入口
    public MapData Generate(int heatLevel)
    {
        // 1. 根据热度调整地图参数
        AdjustParamsByHeat(heatLevel);
        // 2. BSP 切割
        BSPNode root = SplitSpace(new Rect(0, 0, mapWidth, mapHeight), 0);
        // 3. 生成房间
        List<Room> rooms = CreateRooms(root);
        // 4. 连接走廊
        ConnectRooms(root);
        // 5. 分配房间类型
        AssignRoomTypes(rooms);
        // 6. 生成敌人和奖励
        PopulateRooms(rooms, heatLevel);
        return new MapData(rooms, corridors);
    }
}
```

public class DepthScaler : MonoBehaviour
{
    // 根据房间深度返回敌人属性倍率
    public EnemyScaleData GetEnemyScale(int depth)
    {
        return depth switch
        {
            <= 2 => new EnemyScaleData(hp: 1.0f, dmg: 1.0f, count: 2),
            <= 5 => new EnemyScaleData(hp: 1.2f, dmg: 1.15f, count: 3),
            <= 8 => new EnemyScaleData(hp: 1.5f, dmg: 1.3f, count: 4),
            _    => new EnemyScaleData(hp: 2.0f, dmg: 1.5f, count: 5),
        };
    }

    // 根据房间深度返回奖励品质权重
    public RewardWeights GetRewardWeights(int depth)
    {
        return depth switch
        {
            <= 2 => new RewardWeights(common: 80, uncommon: 18, rare: 2),
            <= 5 => new RewardWeights(common: 60, uncommon: 35, rare: 5),
            <= 8 => new RewardWeights(common: 40, uncommon: 45, rare: 15),
            _    => new RewardWeights(common: 20, uncommon: 50, rare: 30),
        };
    }
}

## 十一、风险与应对

| 风险 | 应对方案 |
|------|---------|
| BSP 生成地图不好玩 | 预制部分房间模板，混合使用 |
| 热度系统平衡难调 | 持续收集玩家反馈，数据驱动调整 |
| 遗物组合出 bug 级 build | 设置数值上限，单项属性不超过 ×3 |
| 开发周期拖延 | 严格控制范围，EA 先上 |
| 手感不好 | 花足够时间打磨，这是独立游戏生死线 |
